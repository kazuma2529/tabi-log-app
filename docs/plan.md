# 実装計画書：旅ログ

この計画書は、`docs/requirements.md` の要件と `images/` フォルダ内の完成イメージをもとに、開発初心者でも着手しやすい順番で実装を進めるためのものです。

RevenueCat、App Store 申請、EAS Build、本番課金設定などの初めてだと詰まりやすい作業は、アプリ本体の主要機能が動いてから最後の方で対応します。

---

## 0. 開発時の前提ルール

- 開発を始める前に必ず `docs/requirements.md` とこの `docs/plan.md` を確認する。
- 完成イメージや参考画像は、ユーザーから「画像を確認して」「参考画像を見て」など明示的な依頼があったときに確認する。
  - `images/app-image.png`
  - `images/app-image2.png`
- 旅ノート、紙、地図、スタンプ、手帳のような温かい世界観を維持する。
- 主なアクセントカラーは、完成イメージに近い teal 系と gold/brown 系を基本にする。
- 下部タブ中央の丸いプラスボタンは、アプリの主要導線として早い段階で再現する。
- 実装後は、型チェック、lint、既存テストのようなコードに閉じた確認を自律的に行う。アプリ起動、画面表示、スクリーンショット、参考画像比較、主要フロー操作はユーザーから明示的な依頼があったときに行う。
- `docs/plan.md` は進捗管理にも使う。大きな工程が完了したらチェックを更新する。

## 1. 開発方針

### 1.1 最初に作るもの

最初は「課金なし」「App Store 申請なし」「RevenueCat なし」で、ローカルだけで動くアプリを作る。

優先するもの：

- Expo アプリの土台
- 画面遷移
- 下部タブと中央プラスボタン
- 国マスタ
- SQLite 保存
- 訪問記録追加
- ホーム、地図、日記、統計、バケットリスト
- 参考画像に近いデザイン

後回しにするもの：

- RevenueCat 連携
- App Store Connect 設定
- 実機課金テスト
- TestFlight
- App Store 申請
- iPad 最適化
- データエクスポート
- ウィジェット

### 1.2 技術方針

- Expo SDK 54 を前提にする。
- iOS 専用アプリとして設計する。
- ルーティングは Expo Router を使う。
- データ保存は Expo SQLite を使う。
- 写真はアプリ内ストレージに保存し、SQLite にはファイルパスを保存する。
- 課金状態は最初は開発用の仮フラグで扱い、最後に RevenueCat へ差し替える。
- RevenueCat が必要になるまでは Expo Go で確認できる範囲を優先する。

### 1.3 課金モデル

`docs/requirements.md` 5章で定義されている通り、以下の課金モデルを採用する。Phase 13 以降の実装はすべてこの前提に従う。

- 種別：買い切り（非消耗型 / Non-Consumable）
- 価格：980円
- 解放する機能：以下の 2 つのみ
  1. 写真無制限（無料版は1訪問10枚まで）
  2. 年別分析機能
- サブスクリプション、消耗型課金、広告、外部決済は導入しない。
- 復元ボタンは必須。買い切りのため、機種変更や再インストール時に必ず復元できることを保証する。
- 一度購入したら永続的に有効。アプリ側で期限管理は持たない。

実装上の含意：

- App Store Connect では「非消耗型 In-App Purchase」として課金商品を 1 つだけ登録する。
- RevenueCat の Entitlement は `premium` 1 種類だけで足りる。
- 想定 Product ID：`com.tabilog.premium`（最終的な Bundle ID 確定時に合わせて調整）。
- アプリ内で課金状態を参照する箇所は `isPremium`（boolean）に集約する。

## Phase 13：有料機能の仮実装

目的：RevenueCat の前に、有料機能の画面と制御だけ先に作る。

このフェーズの `isPremium` は、最終的に「買い切り 980円 を購入済みかどうか」を表す（`1.3 課金モデル` 参照）。Phase 15 で差し替えやすくするため、判定ロジックを 1 か所に集約することを最優先する。

- [x] `isPremium = false` のとき、写真は1訪問10枚までにする。
- [x] `isPremium = true` のとき、写真を無制限扱いにする。
- [x] 年別分析を `isPremium = true` のときだけ表示する。

## Phase 14.5：オンボーディング画面を作る

目的：最初の新規ユーザーがアプリを開いた瞬間に、「これは旅行計画アプリではなく、人生で訪れた国を写真と思い出と一緒に残す旅ノートである」と直感的に理解できる 6 枚構成のオンボーディングを作る。

このフェーズは RevenueCat 連携より前に行う。課金導線や購入案内は入れず、アプリ本体の記録体験に自然につなげることを優先する。

参考画像：

- `images/1-2-onboarding.png`
- `images/3-4-onboarding.png`
- `images/5-6-onboarding.png`

### 14.5.1 参考画像から読み取るデザイン方針

- 6 枚すべてを、紙・手帳・旅ノート・地図・スタンプ・パスポートの世界観で統一する。
- 背景は真っ白にせず、古い紙のような淡いクリーム色、薄いシミ、ざらつき、控えめな水彩モチーフで画面全体を包む。
- アクセントは既存アプリと同じ teal 系、gold/brown 系を基本にする。主ボタンは gold/brown、最終スライドの「はじめる」は teal 系にする。
- 上部には「スキップ」を置き、下部にはページドットと CTA ボタンを置く。ページドットは現在ページだけ teal、その他は薄い beige/brown にする。
- タブバーは表示しない。オンボーディングはアプリ本体とは別の全画面体験にする。
- コピーは必ず React Native のテキストとして描画し、生成画像内に日本語コピーを焼き込まない。画像生成の文字崩れを避け、アクセシビリティと修正しやすさを守る。
- 生成画像は「背景 + ヒーロービジュアル」として使い、国名、日付、都市、ボタン、ページドットなど正確性が必要な UI はコードで重ねる。
- iPhone の Dynamic Island / Safe Area を避ける。上部ビジュアルは大きく見せつつ、下部のタイトル・説明・ボタンが小さい端末でも隠れない余白を確保する。

スライド2：訪問国数（カウントの楽しさ）

- タイトル：「今までに、何カ国行きましたか？」
- 説明：「自分が訪れた国を数えて、振り返ろう。気づけば、こんなに色んな場所に行っていた。」

スライド3：国ごとに記録

スライド4：写真とメモで思い出を残す

- ビジュアル：メモカード群（📖 学んだこと / 🍜 食べ物 / 😯 カルチャーショック / ⭐ 次回やりたいこと）と、旅写真サムネイル。
- 役割：「ちゃんと記録できる」価値を一番手厚く見せる。

スライド5：2D地図で旅の足跡

スライド6：行きたい国も + スタート

- 役割：未来側の機能を最後に置きつつ、自然にアプリ開始へ着地させる。

### 14.5.3 画像生成方針

- Image Generation スキルの built-in 画像生成を使い、6 枚それぞれに最適化した縦長の背景画像を作る。
- 生成後の画像は `$CODEX_HOME` などの一時場所に置いたままにせず、必ず `assets/images/onboarding/` 配下へ保存する。
- ファイル名は `onboarding-1.png` から `onboarding-6.png` を基本にする。
- 各画像は iPhone 縦長画面向けに、9:19.5 前後の比率で作る。`ImageBackground` で `cover` 表示しても重要な要素が切れないよう、中央に主役、上下に余白を持たせる。
- 画像には日本語テキスト、数字、国名、日付、ボタン文言を入れない。これらはコード側で重ねる。
- 背景の主役がコピーやボタンを邪魔しないよう、下部 35% は淡い紙面として残す。
- 生成時のプロンプトは、6 枚で同じ画材感（warm watercolor, parchment paper, travel journal, soft shadows）を維持しつつ、各スライドの主役だけを変える。
- 生成後は、色味が既存テーマの `background`, `accentTeal`, `accentGold`, `textPrimary` と合うかを確認し、必要なら 1 回ずつ狙いを絞って再生成する。

### 14.5.4 保存方式と初回表示判定

- オンボーディング完了フラグは `expo-sqlite/kv-store` を使う。
- AsyncStorage は導入しない。通常の SQLite マイグレーションや `purchases` テーブルにも混ぜない。
- キーは `onboarding.completed.v1` のようにバージョンを含める。将来オンボーディングを大きく作り直したときに `v2` として再表示できるようにする。
- 値は文字列で保存する。例：`true` または JSON 文字列（`{"completed":true,"completedAt":"..."}`）。
- `src/lib/onboarding-storage.ts` のような小さなラッパーに、取得・完了保存・開発時リセットを集約する。
- 新規ユーザー判定は、原則として「`onboarding.completed.v1` が未保存」で判定する。
- 既に訪問記録などの主要データがある既存ユーザーにアップデートで突然オンボーディングが出ないよう、実装時に既存データの有無を確認できる場合は、データありユーザーを完了扱いにすることを検討する。
- 「スキップ」と「はじめる」はどちらも完了フラグを保存し、ホームへ遷移する。

### 14.5.5 実装方針

- `app/onboarding.tsx` または `app/onboarding/index.tsx` を追加し、`app/_layout.tsx` の Stack に登録する。
- ルート起動時に完了フラグを読み、未完了ならオンボーディングへ、完了済みなら通常のタブ画面へ進める。
- 読み込み中は紙色背景の最小ローディングにして、ホームが一瞬見えてからオンボーディングへ飛ぶちらつきを避ける。
- スライド定義は `src/features/onboarding/slides.ts` などに集約し、タイトル、説明、ボタン文言、背景画像、重ねるサンプル UI の種類をデータ化する。
- ページ送りは横スワイプできる `FlatList` か `ScrollView` の paging を使う。ボタンでも次へ進める。
- サンプル UI はコードで作る。例：大きな `32カ国`、国旗カード、訪問履歴カード、メモカード、ミニ世界地図、バケットリストカード。
- 既存の `colors`, `spacing`, `radius`, `typography` を使い、オンボーディングだけが別アプリのように見えないようにする。
- タイトルと説明の行間、下部ボタンまわりの余白を固定しすぎず、小さい端末でも文字が切れないようにする。
- 「スキップ」は右上の軽いテキストボタンにし、主 CTA より強くしない。
- 最終スライドだけ CTA を `はじめる` に変え、色も teal 系にしてアプリ開始の意味を明確にする。
